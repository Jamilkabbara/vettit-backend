## Findings

### Finding 1 — Bug I: Targeting Brief MD silently drops countries

- **Severity:** P0
- **Format(s):** Targeting Brief MD
- **Mission(s):** `a520d873…` (22 → 5), Bali (1 → 1, no drop)
- **Symptom (verified via Supabase):**
  - `mission.targeting.geography.countries` = `["US","GB","CA","AU","DE","AE","SA","KW","QA","BH","OM","EG","JO","LB","MA","TN","PS","SY","IQ","YE","DZ","LY"]` (22 ISO-2 codes).
  - `mission.targeting_brief.meta_ads.locations` = `["United States","United Kingdom","Canada","Kuwait","Egypt"]` (5 friendly names).
- **Root cause:** `src/routes/results.js` `/export/targeting-brief` reads `meta.locations` from the AI-generated `targeting_brief` JSONB. The brief generator narrows the list when constructing media-buy specs; the user expected the brief to reflect the full research scope.
- **Proposed fix:** Source the country list directly from `mission.targeting.geography.countries`. Render ISO-2 codes through a `countryName()` map. Never silently drop — if the AI brief had a narrower recommended subset, render it as a *separate* section ("AI-recommended priority markets") with a clear label, while still listing all researched countries above.
- **Verification plan:** Re-export targeting brief for `a520d873…` and confirm 22 countries listed.
- **Status:** TBD

### Finding 2 — Bug B: Screener questions get tautological per-question insights

- **Severity:** P1
- **Format(s):** PDF, PPTX, XLSX, JSON
- **Mission(s):** `a520d873…` Q1 — confirmed tautology
- **Symptom (verified via Supabase):** `insights.per_question_insights[0]` headline = *"All five respondents are active mobile game developers, making the sample directly qualified."* — tautological; the screener filter forced this exact outcome.
- **Cross-mission contrast:** Bali mission Q1 insight is well-formed (*"The screener cleanly split creators and brand managers into two equal groups, but only creator-side voices completed the survey"*) — explains the funnel rather than restating it. Suggests the AI generator can produce useful screener insights when there's a meaningful funnel; tautology only emerges when 100% qualify.
- **Root cause:** No screener-aware override in the export-rendering layer. The renderer copies `per_question_insights` verbatim regardless of whether the question is a screener or a content question.
- **Proposed fix:** When rendering per-question insight for `q.isScreening === true`:
  - **If 100% qualified:** replace insight with sample composition note: *"Sample profile: N of N respondents qualified (100% qualifying rate). Screener used to filter on: <criterion>."*
  - **If <100% qualified:** keep the AI insight (it's likely useful, like the Bali case).
- JSON should keep the raw insight for data fidelity but flag with `is_screener_insight: true`.
- **Status:** TBD

### Finding 3 — Bug H read-time: Schema drift between question options and response distribution

- **Severity:** P0
- **Format(s):** PDF, PPTX, XLSX, JSON, CSV (all show drifted data, none flag it)
- **Mission(s):** `a520d873…` Q4 — 3/5 personas selected `"Rewarded video ads"` (not in schema)
- **Symptom (verified via Supabase):** Q4 schema lists 5 options including `"Interstitial and rewarded video ads"`. Of 5 personas, 2 used the canonical option; 3 used a non-canonical short form `"Rewarded video ads"`. **No persona selected `"Exclusive story content or level packs behind a paywall"` (the 5th canonical option).**
- **Root cause:** Suspect upstream — either (a) the AI persona-simulator was given a different option list at generation time, or (b) the question's `options[]` were edited in the UI after responses were generated. Either way, the export pipeline today shows the drifted distribution silently.
- **In-scope fix (this pass):**
  - **Read-time integrity check:** for each question, verify every `aggregatedByQuestion[qid].distribution` key exists in `question.options[]`. If not, append to `_integrity_warnings: [{ question_id, type: 'unknown_distribution_key', drifted_keys, schema_options }]`. Surface in:
    - JSON: top-level `_integrity_warnings` array (always present, may be empty).
    - PDF: appendix page "Data Integrity Notes" if warnings present.
    - PPTX: appendix slide if warnings present.
    - XLSX: hidden "Data Integrity" sheet (not in tab strip; users can find via "View hidden sheets").
    - CSV: trailing comment rows (deferred — frontend pass).
    - Targeting Brief MD: not applicable (brief doesn't render question data).
- **Out-of-scope (deferred):**
  - Backfill SQL audit across all completed missions — write but don't run.
  - Fixing the upstream simulator pipeline so it reads exclusively from `mission.questions[].options`.
- **Status:** TBD

### Finding 4 — Bug H write-time: Question PATCH guard

- **Severity:** P0
- **Format(s):** n/a — server route
- **Symptom:** No current guard prevents editing `mission.questions` after responses exist. Question text/type/options can drift from what the simulator saw.
- **Proposed fix:** In `src/routes/missions.js` PATCH handler, before applying any change to `questions`, `targeting`, or `respondent_count`, query `mission_responses` for that mission. If `count > 0`, return:
  ```json
  HTTP 409 Conflict
  { "error": "schema_locked_after_responses",
    "message": "Cannot edit questions after responses generated; re-run mission to regenerate." }
  ```

### Finding 5 — Minor 1: Date semantics inconsistent across formats

- **Severity:** P2 (mechanical; cheap to fix)
- **Format(s):** PDF, PPTX, XLSX, JSON, Targeting Brief MD
- **Symptom:**
  - PDF cover: `"Generated April 24, 2026"` — uses today's render date (correct semantically) but labelled ambiguously.
  - PPTX cover: `"Date: April 29, 2026"` — also render date, ambiguous label.
  - XLSX cover: shows BOTH `"Completed at"` and `"Report date"` — correct distinction (best in show).
  - Targeting Brief: `"Generated by VETT AI · ${paid_at}"` — uses `paid_at`, mislabelled as "Generated".
  - JSON: only `exportedAt`, no `mission_completed_at`.
- **Proposed fix:** Shared `getReportMetadata(mission)` util returning two distinct timestamps + labels:
  - `mission_completed_at` (data freshness)
  - `report_generated_at` (document freshness)
- Apply consistently to PDF cover, PPTX cover, XLSX cover, JSON top-level metadata, Targeting Brief frontmatter. **XLSX already has both** — keep its labelling, propagate to the others.
- **Status:** TBD

### Finding 6 — Bug F + G: CSV multi-select percentages + count column

- **Severity:** P1 (claimed) — **unable to statically reproduce in this session**
- **Format(s):** CSV (frontend, `src/pages/ResultsPage.tsx`)
- **Mission(s):** `a520d873…` claimed 20/20/20/16/8 vs. canonical 100/100/100/80/40
- **What I checked:**
  - Frontend `ResultsPage.tsx` (origin/main version) computes `multi_select` percentage as `Math.round((count / total) * 100)` where `total = respondentCount || sum(values) || 1`.
  - For `a520d873…`: `respondentCount = 5`, `count = 5/5/5/4/2`. Expected output: `100/100/100/80/40` — matches canonical, contradicts the prior-audit symptom.
  - Single-choice computes `total = sum(values) || respondentCount` — for Q1 (count=5, sum=5), percentage=100% — also matches canonical.
- **Hypothesis on prior audit:** the symptom was observed against an older deploy *before* a percentage-fix landed; or against a different code path I couldn't find statically (e.g., a CSV download that still uses an older derivation path).
- **Decision:** Cannot ship a fix for what cannot be reproduced. Flagged for **live verification by Jamil** post-Phase-0.1 deploy:
  1. Open mission `a520d873…` results page.
  2. Click "Download CSV".
  3. Open in any plain text editor (NOT Excel — it auto-formats).
  4. Q3 row percentages should be `100%, 100%, 100%, 80%, 40%`. If not, CSV bug is still real and a follow-up frontend pass is needed; capture the actual file and the percentages observed and open a follow-up ticket.
- **Status:** DEFERRED — needs live reproduction; documented as the only audit gap

## Architectural notes / decisions made

1. **Single integrity util** — `src/services/exports/integrity.js` exports `buildIntegrityWarnings(mission, agg)` returning `{ schema_drifts: [], option_overlaps: [] }`. All exporters consume the same util; warnings are surfaced format-appropriately (JSON array, PDF appendix page, PPTX appendix slide, XLSX hidden sheet).
2. **Single date util** — `src/services/exports/reportMetadata.js` exports `getReportMetadata(mission)` returning `{ mission_completed_at, mission_completed_label, report_generated_at, report_generated_label }`. Used by every exporter's cover/header.
3. **ISO-2 → name mapping** — `src/services/exports/iso3166.js` ships a focused map (the 200 most likely VETT geographies — full ISO 3166-1 alpha-2 list). One source of truth, used by Targeting Brief and any future country-aware export.
4. **Screener override util** — `src/services/exports/screenerInsights.js` exports `getSampleCompositionNote(question, agg)` returning a string. Renderers call it conditionally on `q.isScreening`.
5. **Did NOT touch CSV** — frontend lives in `vett-platform/src/pages/ResultsPage.tsx`. Cannot statically reproduce the prior-audit Q3 percentages bug; deferred to a frontend follow-up. The integrity-warnings strategy (trailing comment rows) is also frontend, deferred.
6. **Did NOT introduce new dependencies** — uses existing `pdf-v2` (Phase 0), `pptxgenjs`, `exceljs`, hand-built JSON / Markdown.
7. **Did NOT decommission `pdf.js` (legacy)** — Phase 0 left it as `PDF_LEGACY=1` rollback. This pass keeps that contract.
